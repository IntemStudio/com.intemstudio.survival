## Goal

1주차의 목표는 데모 개발을 시작하기 전에 **데모에 넣을 것과 뺄 것**을 확정하는 것이다. 이번 주에는 큰 기능 구현보다 결정, 기준선 테스트, 작업 목록 축소를 우선한다.

주가 끝날 때는 `Plan_Release_Roadmap.md`의 `Demo Must`가 현실적인 크기로 정리되어 있어야 한다. 이후 2주차부터는 이 결정에 맞춰 무기 성장, 보스/특수몹 체감, 게임오버 통계 같은 핵심 구현에 들어간다.

---

## Success Criteria

- 데모 기본 모드를 확정한다: **아레나 MVP** 또는 **기존 서바이벌 런**.
- 서바이벌 런 길이를 유지 기준으로 정한다: **10~20분 압축형** 또는 **30분 정식형**.
- 데모에서 인벤토리를 어떻게 노출할지 확정한다: 숨김, 제한 공개, 핵심 기능 중 하나.
- 데모 무기 풀 후보를 10~15종으로 줄인다.
- 보스/엘리트/특수몹 중 데모에 고유 체감을 줄 대상을 정한다.
- 골드 UI를 유지할지, 숨길지, 최소 소비처를 둘지 결정한다.
- F5 메인 루프 기준선 테스트 결과를 기록한다.
- 2주차에 바로 구현할 `Demo Core` 작업 3~5개를 확정한다.

## Decisions Needed

### D01 - 데모 기본 모드와 런 구조

| 선택지 | 장점 | 리스크 |
|--------|------|--------|
| 아레나 MVP | 웨이브 단위로 목표가 명확하고 데모 반복이 빠르다. | 기존 30분 밸런스와 다른 축의 QA가 필요하다. |
| 10~20분 압축 서바이벌 | 현재 생존 루프를 짧게 보여 줄 수 있다. | 아레나와 차별되는 데모 핵심이 약할 수 있다. |
| 30분 정식 서바이벌 | 현재 밸런스 구조와 잘 맞는다. | 데모 유저와 QA에 부담이 크다. |

**결정:** 기본 데모는 **아레나 MVP**로 진행한다. 기존 서바이벌은 로비에서 선택 가능한 비교 모드로 유지한다.

### D02 - 인벤토리 노출

| 선택지 | 장점 | 리스크 |
|--------|------|--------|
| 숨김 | 데모 핵심이 명확하다. | 구현된 인벤토리 가치가 드러나지 않는다. |
| 제한 공개 | F6/테스트 빌드에서만 보여 줄 수 있다. | 공개 데모와 테스트 빌드가 갈라진다. |
| 핵심 기능 | 차별점이 된다. | 레벨업 무기와 장비 무기 정책 충돌을 해결해야 한다. |

**결정:** 미정

### D03 - 골드 정책

| 선택지 | 장점 | 리스크 |
|--------|------|--------|
| 유지 | 이미 구현된 HUD와 보상을 살린다. | 소비처가 없어 의미가 약하다. |
| 숨김 | 데모 UI가 단순해진다. | 드랍 보상의 일부가 사라진다. |
| 최소 소비처 | 자원의 의미가 생긴다. | 구현 범위가 늘어난다. |

**결정:** 미정

| 항목 | 확인 내용 | 결과 |
|------|-----------|------|
| F5 로비 | 실행 후 로비 표시, 설정 apply, 게임 시작 | 미확인 |
| 시작 무기 선택 | 게임 시작 후 무기 선택 UI 표시와 선택 가능 여부 | 미확인 |
| 메인 루프 | 몹 스폰, 경험치 수집, 레벨업, 무기 선택 | 미확인 |
| 패배 | 체력 0 이후 게임오버와 재시작 가능 여부 | 미확인 |
| 클리어 | 30분 또는 pace 조정 기준 클리어 시퀀스 | 미확인 |
| F6 테스트 | 몹 스폰, 무기 Equip, 더미 피해 테스트 | 미확인 |
| 문서 정합성 | Roadmap, Wiki, Backlog의 데모 범위 충돌 여부 | 미확인 |

## Risks

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 데모 범위를 너무 크게 잡음 | 2~4주차 계획이 밀림 | Demo Must를 20개 이하로 유지 |
| 인벤토리 노출 결정 지연 | 무기 성장과 UI 정책이 흔들림 | 1주차 안에 숨김/제한/핵심 중 하나 선택 |
| 무기 후보가 너무 많음 | 폴리싱 품질 저하 | 데모 풀 10~15종 제한 |
| 골드 소비처 욕심 | 1주차 스코프 초과 | 정책 결정만 하고 구현은 2주차 이후 |
| 30분 런 QA 부담 | 기준선 확인이 늦어짐 | `balance_pace_multiplier`로 압축 테스트 허용 |
